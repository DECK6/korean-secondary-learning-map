# 구축 로드맵

## 원칙

전체 교과를 한 번에 생성하지 않는다. 먼저 공식 인벤토리와 의미 계약을 고정하고, 수직 슬라이스에서 데이터·온톨로지·화면을 함께 검증한 뒤 교과군을 확장한다.

중학교와 고등학교는 독립 릴리스 열차로 운영한다. `middle`, `high`, `bridges`는 각각 schema·manifest·검수 상태를 가지며, 통합 화면은 검증된 세 버전을 조합한다.

## D0 — 설계 동결

산출물:

- 아키텍처와 데이터 계약
- 역량 질문과 금지 규칙
- 공식 자료 목록과 권리 확인 계획
- K–12 코어 분리 결정 기록

완료 조건:

- 중학교+고등학교 전체를 수용하는 모델인가?
- 두 학교급을 독립 빌드·검수·릴리스할 수 있는가?
- 국가 과목과 학교 개설 과목이 분리되는가?
- 추천과 공식 규정이 분리되는가?
- 초등 P3 식별자를 보존하는가?

## S0 — 공식 인벤토리

작업:

1. 교육부·NCIC의 총론, 중학교, 고등학교, 교과별, 선택·전문 교과 별책을 문서 단위로 수집한다.
2. 파일 해시, 첨부 번호, 페이지 수, 접근일, 권리 표시를 기록한다.
3. 학교급·교과군·과목·과목 범주·성취기준 코드 인벤토리를 추출한다.
4. 원문 대량 재수록 없이 코드와 locator를 검증한다.

완료 조건:

- 정확한 과목·성취기준 개수는 이 단계가 끝난 뒤 처음 발표한다.
- 모든 `official-source-checked` 레코드는 직접 문서와 locator를 가진다.

## M1 — 수학 수직 슬라이스

범위:

- 중학교 수학 전체 구조
- 고등학교 수학의 공통·선택 과목 구조
- 기존 초등 5~6학년 수학과의 전이 관계

이유:

- 선수 관계와 심화 관계가 분명해 관계 의미론을 시험하기 좋다.
- 고등학교 과목 분기와 선택 묶음을 한 교과군 안에서 검증할 수 있다.

완료 조건:

- 공식 코드 앵커 100% 검증
- 모든 주제에 증거·평가 질문 존재
- 선수·전이·과목 관계가 근거와 검토 상태를 보존
- SCQ-01~20 fixture 통과
- 교과 전문가 검토 상태를 자동 검증과 분리해 공개

## M2 — 중학교 공통 기반 확장

우선순위:

1. 국어, 영어
2. 과학, 사회, 역사
3. 기술·가정, 정보, 도덕
4. 체육, 음악, 미술
5. 선택 교과와 창의적 체험활동 연계

교과별 workstream을 독립 생성하고 통합 빌더는 근거 없는 교과 간 간선을 만들지 않는다.

## H1 — 일반 고등학교 공통·선택 확장

작업:

- 공통 과목 전체
- 일반 선택, 진로 선택, 융합 선택 전체
- 교양 및 계열 선택 과목
- 과목 관계, 선택 묶음, 예시 경로
- 고교학점제 `CreditRule` 버전 관리

완료 조건:

- 과목 범주별 인벤토리 완전성 검증
- 학교 개설 여부 비추론 게이트 통과
- 경로 추천의 공식 요건 오인 방지 UI 통과

## H2 — 전문·직업계 확장

작업:

- 전문 공통, 전공 일반, 전공 실무 과목 구조
- 프로그램·교과군·과목의 적용 범위
- 일반 과목과 전문 과목 간 연결
- 직업계 전문가 검토 워크플로

전문 교과를 일반고 경로에 임의로 섞지 않고 `ProgramType`과 적용 범위를 통해 탐색 가능하게 한다.

## U1 — 통합 탐색 화면

구현 순서:

1. L0~L3 집계 지도
2. 과목 상세와 L4~L5 하위 그래프
3. 초→중→고 전이 지도
4. 과목 비교
5. 경로 설계기
6. 선택적 학교 개설 데이터 조인

성능 기준:

- 첫 화면은 전체 세부 주제 그래프가 아닌 집계 그래프를 로드한다.
- 선택한 과목·클러스터 하위 그래프만 지연 로딩한다.
- 검색 결과는 학교급·과목 범주·검토 상태를 함께 표시한다.
- 키보드 탐색, 모바일, 저사양 기기에서 핵심 정보 접근이 가능해야 한다.

## 릴리스 게이트

| 게이트 | 검증 |
| ---: | --- |
| G1 | 공식 문서·해시·locator·권리 메타데이터 |
| G2 | JSON Schema와 참조 무결성 |
| G3 | 식별자 안정성·중복·replacement 정책 |
| G4 | 관계 의미·사이클·적용 범위·파생 규칙 |
| G5 | OWL/RDF 파싱, SHACL, SPARQL 역량 질문, 적대 fixture |
| G6 | 결정적 빌드와 manifest SHA-256 |
| G7 | 모델 후보·내부 검토·교과 전문가·현장 검토 상태 분리 |
| G8 | 국가 과목/학교 개설, 공식 규정/추천 경로 오인 방지 UI |
| G9 | 초등 P3 호환성과 학교급 전이 질의 |

자동 게이트 통과는 교육부 승인, 교과 전문가 승인, 현장 타당성, 권리 허가를 뜻하지 않는다.

### 학교급별 승격 규칙

- 중학교 게이트 통과만으로 고등학교 릴리스를 승격하지 않는다.
- 고등학교 게이트 통과만으로 bridge를 재생성·승격하지 않는다.
- bridge는 참조한 중·고 릴리스 ID와 해시가 그대로일 때만 유효하다.
- 통합 bundle은 세 하위 릴리스의 내용을 다시 쓰지 않고 버전·해시·호환 상태만 묶는다.
- 별도 저장소 분리는 독립 팀, 독립 배포 주기, 접근 권한 분리가 실제 요구가 된 시점에 결정한다.

## 후보 릴리스 이후 작업

S0·M1·M2·H1·H2·U1의 기술 산출물은 `v0.3.0-candidate`에서 전 교과 범위로 구현하고, 중학교 주제 분해 밀도와 고등학교 범위별 비교를 보완했다. 이후 작업은 자동 생성량 확대가 아니라 검토 승격이다.

1. 교과·직업계 전문가가 과목 판별, 영역, 세부 주제와 관계 후보를 표본→전수 순으로 검토한다.
2. 문서별 재사용 조건을 확인하고 권리 HOLD 해제 여부를 별도 결정한다.
3. 초등 P3 릴리스와 중학교 릴리스를 모두 고정하는 별도 elementary-secondary bridge를 만든다.
4. 실제 학교 개설 정보가 필요한 경우 국가 `Course`와 섞지 않고 선택 모듈로 조인한다.
5. 검토 기록이 축적되면 candidate를 reviewed 릴리스로 승격하되 기존 manifest와 ID를 보존한다.
